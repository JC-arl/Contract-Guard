# LLM 양자화 수준 선택 기능 및 정확도 비교 스윕 계획

## 배경

현재 Contract-Guard는 Ollama의 `exaone3.5:7.8b`(기본 Q4_K_M) 한 가지 양자화 수준에 고정되어 있다. 양자화 수준을 바꾸려면 `.env` 파일의 모델명을 직접 수정하고 백엔드를 재시작해야 한다. 이로 인해 두 가지 문제가 있다.

1. **일반 사용자 관점**: 저사양 PC(내장 GPU·일반 노트북) 사용자는 느린 응답 때문에 서비스 사용이 어렵고, 고사양 PC 사용자는 더 정확한 결과를 선택할 수 있음에도 선택지가 없다.
2. **캡스톤 발표 관점**: "양자화 수준이 분석 정확도와 속도에 어떤 영향을 주는가"를 정량적으로 제시하기 어렵다. 한 번에 한 수준만 실험 가능하며 결과를 체계적으로 수집할 장치가 없다.

## 목표

1. **사용자 기능**: 업로드 화면 드롭다운으로 양자화 수준을 선택해 분석 실행. 선택값은 업로드 요청에 함께 전송되며, 백엔드가 해당 수준의 모델로 응답.
2. **스윕 기능**: 검증 스크립트가 여러 양자화 수준을 순회하며 동일한 검증셋으로 정확도·소요시간을 측정하고 비교 결과를 저장.
3. **하위호환**: 기존 `.env`의 `OLLAMA_MODEL_NAME` 설정이 그대로 동작하도록 유지.

## 비목표

- 커스텀 Modelfile을 이용한 재양자화 (Ollama 레지스트리 기본 태그만 사용).
- 임베딩 모델 양자화 (LLM만 대상).
- CPU/GPU Docker 프로파일 구조 변경.
- 프론트엔드 전역 상태 관리 라이브러리 도입.

## 지원하는 양자화 수준

총 5개 수준을 비교 대상으로 한다. 양자화가 낮을수록 속도는 빠르고 정확도는 떨어지며, 높을수록 반대다.

| 사용자 라벨 | Ollama 태그 | 권장 환경 |
| --- | --- | --- |
| 빠름 (저사양 PC 권장) | `q2_K` | 내장 그래픽·저사양 노트북 |
| 빠름·균형 | `q3_K_M` | 중간 사양 |
| 균형 (기본) | `q4_K_M` | 표준 — 현재 기본값과 동일 |
| 정확·균형 | `q5_K_M` | 상위 사양 |
| 정확 (고사양 권장) | `q8_0` | 고성능 GPU·대용량 RAM |

기본 선택은 `균형 (q4_K_M)`이며, 사용자는 드롭다운에서 원하는 수준으로 변경할 수 있다.

## 사용자 동작 흐름

1. 사용자가 업로드 페이지에 접속하면 파일 선택 영역 부근에 양자화 수준 드롭다운이 표시된다.
2. 드롭다운의 기본값은 `균형 (기본)`이다. 사용자는 필요 시 다른 수준을 선택할 수 있다.
3. 파일을 선택해 업로드하면 선택된 양자화 수준이 요청과 함께 백엔드에 전송된다.
4. 백엔드는 현재 활성 모델과 요청된 수준이 다르면 모델을 교체한 뒤 분석을 수행한다. 같으면 교체 없이 바로 실행한다.
5. 분석 결과는 기존과 동일하게 결과 페이지로 전달된다.

양자화 수준이 바뀔 때 Ollama가 기존 모델을 언로드하고 새 모델을 로드하는 동안 **첫 응답이 수 초에서 수십 초 지연**될 수 있다. 같은 수준으로 반복 사용할 경우 모델이 메모리에 유지되어 지연이 없다.

## 스윕 기능 (캡스톤 발표용)

검증 스크립트에 두 가지 옵션이 추가된다.

- **단일 수준 실행**: 특정 양자화 수준 하나로 전체 검증을 수행.
- **스윕 실행**: 여러 양자화 수준을 순회하며 동일 검증셋에 대해 각각 정확도와 소요시간을 측정.

스윕 결과는 `data/` 디렉토리에 타임스탬프가 포함된 JSON 파일로 저장되며, 콘솔에도 수준별 비교 표가 출력된다. 수집하는 지표는 다음과 같다.

- 전체 정확도
- 유리(safe) 정확도 / 불리(risky) 정확도
- 놓친 위험(false safe) 건수, 거짓 경보(false risky) 건수
- 총 소요시간, 조항당 평균 소요시간

`temperature=0`과 `seed=42`로 디코딩 변수가 통제되므로, 수준 간 차이는 양자화 가중치 차이에 기인한 것으로 해석할 수 있다.

## 하위호환 정책

- 기존 `.env`에 `OLLAMA_MODEL_NAME=exaone3.5:7.8b`만 지정되어 있으면 **현재 동작 그대로 유지**된다. 양자화 필드가 비어 있는 업로드 요청도 기본 설정으로 처리된다.
- 프론트엔드 드롭다운을 건드리지 않고 업로드하면 기존 사용자가 느끼는 변화는 없다.

## 검증 방법

1. **사전 준비**
   - 5개 양자화 태그를 한 번에 내려받는 스크립트를 실행하여 Ollama 레지스트리에서 사용 가능한 실제 태그 이름을 확인한다.
   - 필요 시 태그 이름 규칙을 문서에 반영한다.

2. **백엔드 동작 점검**
   - 서비스 기동 후 모델 정보 조회 엔드포인트로 현재 활성 모델이 기본값인지 확인.
   - 서로 다른 양자화 수준으로 업로드 요청을 반복해 모델 교체가 1회만 일어나고, 같은 값 재요청 시 교체가 일어나지 않는지 로그로 검증.

3. **프론트엔드 통합 점검**
   - 드롭다운에 5개 옵션이 올바르게 표시되고 기본값이 `균형 (기본)`인지 확인.
   - 서로 다른 수준을 선택해 업로드했을 때 네트워크 요청에 올바른 양자화 필드가 포함되는지 확인.
   - 분석 결과 페이지가 기존과 동일하게 정상 도착하는지 확인.

4. **스윕 결과 점검**
   - 스윕 실행 후 생성된 JSON 파일과 콘솔 비교 표를 검토한다.
   - 기대 신호: `q4_K_M` 이상에서 정확도가 비슷하게 수렴하고 `q2_K`에서 유의미한 저하가 나타나야 한다. 수준 간 정확도가 전혀 다르지 않다면 실제로는 모델 교체가 이뤄지지 않았다는 신호이므로 교차 검증이 필요하다.

5. **회귀 점검**
   - 기본 설정 상태에서 드롭다운을 건드리지 않은 채 업로드했을 때 기존과 동일한 결과가 나오는지 확인.

## 위험 요소와 트레이드오프

- **모델 교체 지연**: Ollama가 모델을 언로드/로드하는 동안 첫 요청은 수~수십 초 느려질 수 있다. 동일 사용자가 같은 수준을 반복 사용할 때는 영향이 없다.
- **디스크 사용량**: 5개 양자화 태그를 모두 내려받을 경우 약 25~35GB의 디스크가 필요하다. 실험 종료 후 불필요한 태그는 수동 정리가 필요하다.
- **동시 요청 경합**: 두 사용자가 서로 다른 수준을 동시에 요청하면 내부 직렬화로 인해 순차 처리된다. 소수 사용자·데모 환경에서는 수용 가능하나 대규모 동시 접속에는 부적합하다.
- **디코딩 통제의 한계**: `temperature=0` · `seed=42`로 디코딩은 결정적이지만, 저양자화 모델은 가중치 자체의 수치 오차로 출력이 흔들릴 수 있다. 결과 해석 시 "동일 디코딩 설정하의 양자화 영향"임을 명확히 한다.
- **Ollama 태그 명명 가정**: 모델명 조합 규칙은 Ollama 레지스트리의 현재 명명에 의존한다. 구현 첫 단계에서 실제 pull 가능 여부로 규칙을 고정한 뒤 코드에 반영한다.

## 기대 효과

- 일반 사용자는 자기 하드웨어 수준에 맞는 속도·정확도 균형을 직접 선택할 수 있다.
- 개발자는 한 번의 명령으로 여러 양자화 수준의 품질을 비교하여 기본값 선정이나 배포 전략의 근거 자료를 확보할 수 있다.
- 캡스톤 발표에서 "모델 크기 축소가 계약서 분석 품질에 미치는 영향"이라는 주제를 정량적으로 제시할 수 있다.
